## 三、Z 进程 — 自动运维巡检

**Z 进程是一个有判断力的运维巡检**，不是傻跑命令。每一步先检查，有问题才修，修完再验。

### 流程

```
检查数据是否最新 → 不是就更新 → 核对数据准确性
       ↓
检查画像/缓存是否正确 → 有问题就刷新
       ↓
检查工具健康 → 有问题就修复 description
       ↓
记录所有发现的问题 → 输出巡检报告
```

### 阶段 1：数据新鲜度检查

**目标**：所有本地数据是否覆盖到最新月份？

```bash
# 1a. dameng_mirror 诊断 — 哪些表缺最新月份？
python3 ops/scripts/sync/sync_mirror.py
# 输出：每张表的最新月份、缺失月份清单

# 1b. 基础数据诊断 — 8 张 F 类表行数是否正常？
python3 manage.py sync_base_data --dry-run
# 输出：远端 vs 本地行数对比
```

**判断**：
- 有缺失月份 → 执行 `ops/scripts/sync/auto_sync.py` 或 `sync_mirror.py --month YYYYMM` 补数据
- F 类行数异常 → 执行 `manage.py sync_base_data` 全量刷新
- 全部正常 → 跳过，进入下一阶段

### 阶段 2：数据准确性核对

**目标**：数据不只是"有"，还得"对"。

```bash
# 五维验证：表存在/行数/月份覆盖/SA覆盖/内容抽查
python3 ops/scripts/sync/sync_mirror.py --verify-only
```

**判断**：
- 行数不一致 → 清理后重拉 `--clean --month YYYYMM`
- SA 覆盖率低 → 可能是上游数据缺失，记录到报告
- 内容抽查不一致 → 比对具体字段，定位是本地还是远端问题

### 阶段 3：画像与缓存检查

**目标**：Redis 里的预缓存是否健康？

```bash
# 缓存交叉验证（结论/基准/画像/维度/实时）
python3 manage.py verify_caches
```

**判断**：
- 结论/基准缺失 → `refresh_conclusions` / `refresh_benchmarks`  刷新
- 画像数量 < 100 → `generate_insights` 补充
- 实时缓存 = 0 → 检查 cron 是否正常、API Key 是否有效
- DRIFT（偏差 1-10%）→ 记录，可能是时间窗口差异
- FAIL（偏差 >10%）→ 标红，清缓存后重新生成

### 阶段 3.5：风险画像审核

**目标**：Redis 中的风险画像缓存是否完整？数据是否准确？

#### 3.5a 数量检查

```bash
# 检查风险缓存 key 数量
python3 -c "
import redis
r = redis.Redis(decode_responses=True)
keys = list(r.scan_iter('risk:*', count=5000))
print(f'风险画像: {len(keys)} 个 key')
if len(keys) < 1000:
    print('⚠️ 风险画像不足 1000，需要重新生成')
else:
    print('✅ 风险画像数量正常')
"
```

**判断**：
- key 数 ≥ 2000（150SA × 14月）→ 正常
- key 数 < 1000 → 执行重新生成：
  ```bash
  python manage.py refresh_conclusions --domain risk --month $(date -d 'last month' +%Y%m)
  python manage.py refresh_conclusions --domain risk --month $(date -d '2 months ago' +%Y%m)
  ```

#### 3.5b 准确性交叉验证（抽样）

从缓存中随机抽 3-5 个服务区的最新月份数据，对照 `dameng_mirror.db` 中 SABFI_SHOPS 源数据验证关键数值：

| 缓存字段 | 对照查询 | 偏差阈值 |
|----------|----------|---------|
| `merchant_warning.商户总数` | `SELECT COUNT(DISTINCT 经营商户) FROM SABFI_SHOPS WHERE 服务区内码=X AND STATISTICS_MONTH=M` | 必须精确一致 |
| `project_risk.项目总数` | `SELECT COUNT(DISTINCT 经营项目名称) FROM SABFI_SHOPS WHERE ...` | 必须精确一致 |
| `project_risk.亏损项目` | `SELECT COUNT(DISTINCT 经营项目名称) FROM SABFI_SHOPS WHERE ... AND 盈利金额<0` | 必须精确一致 |
| `payment_status.记录总数` | 到账明细 API 返回条数 | 允许 ±1 条（API 延迟） |

```bash
# 抽样验证脚本（Z 进程在服务器上执行）
python3 -c "
import redis, sqlite3, json, random

r = redis.Redis(decode_responses=True)
db = sqlite3.connect('data/dameng_mirror.db')

# 随机抽 3 个最新月份的 risk key
keys = [k for k in r.scan_iter('risk:*', count=5000)]
sample = random.sample(keys, min(3, len(keys)))

for key in sample:
    data = json.loads(r.get(key))
    dims = data.get('dimensions', {})
    sid = data.get('server_id')
    month = data.get('month')
    
    # 交叉验证商户数
    cached_cnt = dims.get('merchant_warning', {}).get('data', {}).get('商户总数', -1)
    real_cnt = db.execute(
        'SELECT COUNT(DISTINCT 经营商户) FROM SABFI_SHOPS WHERE 服务区内码=? AND STATISTICS_MONTH=?',
        (sid, month)
    ).fetchone()[0]
    
    match = '✅' if cached_cnt == real_cnt else '🔴 FAIL'
    print(f'{key}: 商户数 缓存={cached_cnt} 实际={real_cnt} {match}')

db.close()
"
```

**判断**：
- 全部 ✅ → 通过
- 有 🔴 FAIL → 缓存与源数据不一致，重新生成该服务区/月份：
  ```bash
  python manage.py refresh_conclusions --domain risk --server-id <sid> --month <month>
  ```

### 阶段 4：工具健康审计

**目标**：68 个注册工具（55 个 LLM 可见）的 description、参数、域路由是否合规。

```bash
# 审计
python3 ops/scripts/audit/audit_tools.py

# 如果有问题，自动修复 description（仅限字符串，不改逻辑）
# 修复前必须 git 保护
git add -A && git commit -m "z: 修复前快照"
# 修复后重跑审计确认 0 Error 0 Warning
```

**判断**：
- Error = 0, Warning = 0 → 通过
- 有 Error/Warning → 按 `ops/工具规范.md` 标准修复 description
- 台账过期 → `audit_tools.py --gen-doc` 重新生成

### 阶段 5：输出巡检报告

每次巡检结果写入报告，记录所有发现：

```markdown
## Z 进程巡检报告 — YYYY-MM-DD

### 数据新鲜度
- dameng_mirror: ✅ 全部覆盖到 YYYYMM / ⚠️ N 张表缺 YYYYMM（已补）
- db.sqlite3: ✅ 8 表行数正常 / ⚠️ 行数异常已刷新

### 数据准确性
- 五维验证: ✅ 全部通过 / ⚠️ N 个问题（明细...）

### 画像与缓存
- 结论: X 个 key ✅ / ⚠️ 缺失已刷新
- 基准: X 个 key ✅
- 画像: X 个 ✅ / ⚠️ 不足 100 已补
- 实时: X 个 key ✅
- **风险**: X 个 key ✅ / ⚠️ 不足 1000 已重新生成

### 工具审计
- Error: 0 / Warning: 0 / Info: N
- 台账: 最新 ✅

### 发现的问题（需人工关注）
- ...
```
